# Architecture

Godot 4.7 project, GL Compatibility renderer. Engine version and renderer are pinned in `project.godot` — do not upgrade without checking these settings still make sense.

## Directory layout

- `scenes/` — one `.tscn` per composable game object (player, platform, level). No loose top-level scenes outside this folder.
- `scripts/` — one `.gd` per node that needs behavior, named after the node/scene it's attached to (`player.gd` on `Player`, not on some unrelated node).
- `assets/` — non-script, non-scene resources, grouped by type (`assets/audio/`, `assets/shaders/`).
- `data/` — `.tres` tuning resources (e.g. `player_tuning.tres`), instances of a `Resource` subclass in `scripts/` (e.g. `gameplay_tuning.gd`). Balance knobs that aren't tied to a single node instance live here rather than as `@export` vars on the script, so they're editable as data without a code change.

## Where information lives

Docs hold only what the code can't express:

- **Rationale anchored to code** (why a node is structured this way, why an ordering matters, why an approach was chosen) lives as a comment in the script, at the line a future edit would break.
- **Cross-scene contracts and conventions** live here, once, as the named patterns below.
- **Everything else** — node lists, signal signatures, tuning values — lives only in the `.tscn`/`.gd` files. Don't restate it in docs; open the code.

When a scene's role changes or a pattern is added/broken, update the catalog entry or pattern here in the same change.

## Patterns

Every scene and script follows these; deviate deliberately and say why in a comment.

1. **Signals up, calls down; `Main` owns spawning.** Gameplay children announce events with past-tense signals (`seed_popped`, `seed_planted`, `bloomed`); `Main` connects to them (duck-typed via `has_signal`, so instances added later are covered automatically) and is the only place that instantiates scenes into the level at runtime. No scene ever touches the tree above itself.
2. **Passive detectables, one sensor poll.** Everything the bug interacts with in the world (`water`, a seedling's `HoverZone`, `seed`, `plot` — group names match the scene) is an `Area2D` that knows nothing about the player. The player's `WaterSensor` polls overlaps once per physics tick and owns all interaction logic; a detectable at most exposes a method or property for the player to call.
3. **UI reaches into gameplay, never the reverse.** HUD elements find `player`/`main` via groups and connect to their signals; gameplay scripts stay UI-agnostic. Each HUD panel is self-contained — its own script, its own toggle input — independent of its siblings.
4. **`PlantData` is the single source of truth for plant facts** (types, colors, combos, display names). Gameplay and UI both read from it; nothing else may hardcode plant data. UI content is *generated* from it (the combo chart builds rows from `PlantData.all_combos()`, icons are extracted from `seedling.tscn` via `PlantIconSource`) rather than hand-authored copies that could drift.
5. **Parallax scenery is screen-anchored, not world-anchored.** Background layers live in `background.tscn` (`ParallaxBackground`/`ParallaxLayer`); layer children are positioned in screen-space coordinates (where they sit when the camera's top-left is at world origin), with `motion_scale` < 1 for depth and `motion_mirroring.x` set to the layer's content tile width — which must exceed the visible world width (1152 / camera zoom 1.6 = 720px) or the wrap shows a gap at screen edges. Scenery pieces themselves (`pine_tree.tscn`, `cloud.tscn`) follow the same scriptless flat-`Polygon2D` convention as `platform.tscn`. A layer can also drift on its own, independent of the camera: `parallax_autoscroll.gd` nudges `motion_offset.x` each frame (additive with the camera-driven `scroll_offset`, so screen-anchoring of layer children is undisturbed) and wraps it at `motion_mirroring.x` so the seam stays invisible. `Clouds`/`CloudsFar` use this for an autonomous rightward drift, each with its own `scroll_speed` (the more distant `CloudsFar` slower than `Clouds`) for extra depth on top of `motion_scale`.
6. **Feedback conventions.** `CPUParticles2D` only — the GL Compatibility renderer has no GPU particles. One-shot SFX get slight `pitch_scale` randomization per play; looping SFX are volume-faded in/out, never hard play/stopped. Visuals are a pure function of state (`_update_visuals()`-style, no visual state stored elsewhere), and transitions animate briefly (~0.15–0.4s — "nothing snaps", per GAMEPLAY.md).
7. **Fired projectiles are the one active exception to passive detectables.** A `WaterDroplet` (`water_droplet.tscn`) carries its own short-lived collision poll because it, not the player, is what's moving through the world during its lifetime — unlike Pattern 2's detectables, it owns its own velocity/gravity integration and its own overlap checks against seedlings and the ground, instead of being polled by a sensor on the player.

## Scene catalog

Orientation only — read the `.tscn`/`.gd` for detail.

- `main_menu.tscn` — the entry point (project main scene): title, how-to-play text, and music credit on an opaque background. On Space it swaps the whole scene to `main.tscn` via `change_scene_to_file` — the level and the menu never coexist, so there is no intro pause gate.
- `main.tscn` — currently *the* single static level (there is no level-loader; per CODING.md, don't build one until a second level exists), entered from `MainMenu`. `main.gd` owns all runtime spawning (popped seeds, planted seedlings, the win overlay) and goal tracking: 4 random goal plants rolled per run, checked off as they bloom, win overlay when all 4 are done.
- `background.tscn` — `ParallaxBackground` (draws behind everything; `CanvasLayer` layer defaults to -100) with four `ParallaxLayer`s, back to front: `Sky` (screen-fixed `motion_scale` 0, a copper-style banded blue gradient of `Polygon2D` strips), `CloudsFar`, `Clouds` (both `cloud.tscn` instances at varied position/scale, pattern 5; `CloudsFar` smaller/more numerous for depth), and `Trees` (soil/grass band plus `pine_tree.tscn` instances at varied position/scale). Instanced once as `Main`'s first child; scroll tracks the active `Camera2D` automatically, no wiring for that part — `CloudsFar`/`Clouds` additionally carry `parallax_autoscroll.gd` (pattern 5) for their own slow autonomous rightward drift, `CloudsFar` slower than `Clouds`.
- `cloud.tscn` / `pine_tree.tscn` — scriptless `Node2D` scenery primitives: a cloud is three overlapping 16-gon `Polygon2D` "circles", a pine is a trunk plus three stacked triangle tiers with origin at the ground line.
- `platform.tscn` / `wall.tscn` — scriptless `StaticBody2D` geometry. Resize instances by scaling the root (collision and visuals scale together); never edit the shape resource per-instance. `wall.tscn` is invisible (collision only, no `Visual` node) — it's the top/bottom level bounds, where no visual is wanted.
- `tree_trunk_wall.tscn` — scriptless `StaticBody2D`, the left/right level bounds: a wide brown tree-trunk visual (stacked bark-toned `Polygon2D` shapes) authored in its natural vertical orientation so instances need no rotation, just positioning.
- `water.tscn` — passive detectable the player can rest on (physical snap) and, within proboscis suck range, draw water from; exposes `get_surface_y()` only.
- `player.tscn` — the flying bug: thrust-based flight (not platformer walk/jump), proboscis-driven water sucking (automatic on proximity alone — the proboscis extends and drinks the moment its tip is close enough to a water surface, no button) and, away from water, firing (button, hold-to-spray) real `water_droplet.tscn` projectiles (pattern 7). Watering a seedling only ever happens via a droplet hit; hovering does nothing to the tank or a plant. Also pollen and seed carrying. All world interaction is via the pattern-2 sensor poll except fired droplets, which self-poll (pattern 7). Everything that mirrors when the bug turns is under `Visual`; collision and camera are deliberately outside it.
- `water_droplet.tscn` — a fired projectile, spawned by `main.gd` on the player's `water_fired` signal (pattern 1); integrates its own gravity, self-polls overlaps each physics tick (pattern 7), and despawns via a `Timer` — a fixed lifetime in flight, shortened on impact so its hit/miss splash gets to play first. Hitting a `seedling`-group area waters it once and plays an absorb splash; hitting anything else (water, platform, wall) is a miss and just splashes. Fire/hit-splash/miss-splash sounds currently reuse placeholder clips (`water_pour.wav`, `seed_plant.wav`) pending dedicated audio assets.
- `enemy.tscn` — gnat-cloud hazard drifting on a leash around its spawn point; steals water and knocks off carried pollen from the player on touch (seeds are never touched).
- `flower_guard.tscn` — a second self-directed hazard alongside the gnat cloud (same active-entity shape as `enemy.gd`), meant to guard whichever flower it's placed next to against point-blank camping (there's no separate flower reference — its own spawn position *is* the guarded point, so placement in `main.tscn` is what ties a guard to a flower). Orbits that spawn point on a fixed circle, building/decaying aggro from player dwell time near it and telegraphing a windup (tint, wing-buzz speed, rising-pitch `WindupBuzz`) that cancels if the player backs off. Once windup completes it dashes (`LUNGE` state) at the player's position captured at that instant — not homing — then eases back onto the orbit circle (`RETURN` state) before resuming patrol; the dash is not cancelable once started. Detects the player via `body_entered` (the root `Area2D`'s `monitoring=true` is on throughout) and, on contact in *any* state — patrol, windup, lunge, or return, not just a connecting lunge — duck-types `steal_water()`/`lose_pollen()` exactly like `enemy.gd` (seeds untouched), playing a distinct `HitSound` (a synthesized percussive thud, `flower_guard_hit.wav`, distinct from both `water_pour.wav` and the gnat's `water_steal.ogg`). Aggro always resets to 0 after the lunge resolves (hit or miss) and `reaggro_cooldown` blocks it from rebuilding for a beat afterward. Five instances are placed in `main.tscn` guarding individual flower clusters; a dedicated lunge whoosh and continuous patrol buzz are still outstanding fit-and-finish.
- `seedling.tscn` — plant lifecycle state machine: `GROWING → BLOOMED → POLLINATED → SEED_GROWING → (pops a seed) → BLOOMED`, so seeds are renewable. `@tool`, so growth/bloom edits preview live in the editor. One `Bloom/<Name>` visual subtree per plant type, named exactly by `PlantData.display_name()` — a lookup contract relied on by `seedling.gd` and `PlantIconSource`. Bloom polygon colors are hand-authored in the `.tscn` to *match* `PlantData` (a deliberate pattern-4 exception; only pollen cues and seed pods are colored from `PlantData` at runtime).
- `seed.tscn` / `plot.tscn` — the passive detectables of the carry-and-plant loop: a loose seed is consumed by the player's poll; a plot's `plant()` emits `seed_planted` for `Main` to spawn the new seedling.
- `hud.tscn` — `WaterMeter` (player water level), `ComboChart` (toggle "1"), `GoalPanel` (toggle "2"). The two panels are child-node-free custom-`_draw()` Controls whose rows are generated per pattern 4 (combos from `PlantData`, goals from `Main`'s roll).
- `win_overlay.tscn` — spawned by `Main` on win rather than pre-placed (one-shot terminal UI); "Play again" reloads the level scene, which re-rolls the goals.
- `scripts/plant_data.gd` (`PlantData`, not attached to a node) — the pattern-4 data table: `PlantType` enum (explicit values — `bloom_type` ints saved in `main.tscn` must keep mapping to the same plants), color tables, `ACCEPTS_POLLEN`, the hybrid combo table.
- `scripts/plant_icon_source.gd` (`PlantIconSource`, not attached to a node) — extracts bloom polygon/color data by instantiating `seedling.tscn` off-tree, so UI icons can never drift from the in-world plant visuals and no live seedling (with its collision/groups) ever backs a decorative icon.

## Known gaps / not yet built

- No level-transition or scoring beyond the single win condition — nothing consumes `water_level` besides watering, and there's no losing state (Step 7's enemies are pressure, not failure).
- No save/settings system.
- Input actions are defined by hand in `project.godot`; there is no in-game rebinding UI.

Update this section as these are built, rather than leaving it to go stale.
